  ---                                                                                                                   
  RÉSUMÉ EXÉCUTIF                                                                                                       
                                                                                                                        
  EASY.Q est une plateforme SaaS de menu digital QR pour la restauration, incluant un chatbot IA, un système de
  commande/paiement Stripe Connect, un KDS (écran cuisine), et des analyses. La stack technique est solide et bien      
  structurée, avec des abstractions appropriées (services, schémas Pydantic, migrations Alembic, audit log RGPD). Les
  416 tests backend et 113 tests frontend passent intégralement.                                                        
                                                            
  Cependant, l'audit révèle 3 vulnérabilités de sévérité critique qui rendraient le déploiement en production dangereux 
  dans l'état actuel :
                                                                                                                        
  1. Modification non authentifiée des profils restaurant — tout acteur peut rediriger les paiements vers son propre    
  compte Stripe en une requête PATCH.
  2. Vérification de signature JWT absente sur /auth/me — toute session utilisateur peut être usurpée sans clé secrète. 
  3. Aucun rate limiting sur les endpoints IA/chat — exposition à une attaque par épuisement de quota Gemini.

  CONSTATATIONS DÉTAILLÉES PAR DOMAINE                      
                                                                                                                        
  a. Architecture et Environnement                          
                                                                                                                        
  Points forts :                                                                                                        
  - Architecture en couches propre : routers → services → models, respectant le principe de responsabilité unique
  - 11 migrations Alembic tracées, schéma évolutif et documenté                                                         
  - PWA correctement configurée (Service Worker, manifest, runtime caching différencié)
  - Redis utilisé comme cache L1 (menu 5 min, session chat 2h, résultats OCR 24h) avec dégradation gracieuse si Redis   
  est hors ligne                                                                                                        
  - Cloudflare R2 avec fallback local sur STORAGE_DIR — bonne résilience                                                
  - Sentry conditionnel (SENTRY_DSN optionnel), Langfuse pour l'observabilité LLM                                       
  - Middleware de logging des requêtes lentes (>500ms) avec X-Request-ID                                                
                                                                                                                        
  Problèmes identifiés :                                                                                                
                                                                                                                        
  [MOYEN] Le cache _GOOGLE_RATING_CACHE (restaurants.py:252) est un dictionnaire Python en mémoire, sans TTL d'éviction 
  et non partagé entre workers. Dans un déploiement multi-worker (Gunicorn/Uvicorn avec --workers N), chaque processus
  maintient sa propre copie. Le cache grossit indéfiniment en mémoire. → Migrer vers Redis avec TTL.                    
                                                            
  [MOYEN] Le cache JWKS (_jwks_cache) dans auth.py:67 est stocké pour la durée de vie du processus, sans TTL. Si Clerk  
  effectue une rotation de clés, les nouvelles clés ne seront reconnues qu'après redémarrage. → Ajouter un TTL de 24h 
  avec rafraîchissement en arrière-plan.                                                                                
                                                            
  [BAS] Dépendance circulaire contournée par import local : from app.routers.orders import _next_pickup_number à        
  l'intérieur de payments.py:140. Ce pattern indique un couplage fort entre les modules payments et orders. → Déplacer 
  _next_pickup_number dans un service partagé.

  c. Intégrité des Données et Précision Fonctionnelle
                                                     
  Points forts :
  - Calcul des montants côté serveur (payments.py:87) : la somme des items est recalculée à partir des données serveur, 
  empêchant la manipulation des prix côté client.                                                                       
  - Minimum Stripe (50 centimes) vérifié avant création du PaymentIntent.                                               
  - AuditLog immuable tracé pour les événements financiers et utilisateurs — base solide pour la RGPD.                  
  - La commande (Order) est créée de façon atomique avec le paiement dans une seule transaction DB.                     
                                                                                                                        
  Problèmes identifiés :                                                                                                
                                                                                                                        
  [MOYEN] Race condition sur les numéros de commande (_next_pickup_number dans orders.py:79) : cette fonction lit le    
  MAX(pickup_number) puis incrémente. Sous charge concurrente, deux requêtes simultanées peuvent obtenir le même numéro 
  de commande. → Remplacer par une séquence PostgreSQL ou un SELECT FOR UPDATE.                                         
                                                            
  [MOYEN] Dans payments.py:254-263, la résolution de l'Order à confirmer lors d'un webhook Stripe                       
  payment_intent.succeeded se fait par table_token + status="pending" + ORDER BY created_at DESC — sans lien direct au
  payment_intent_id. Si deux commandes sont en cours simultanément pour la même table (bug ou retry), la mauvaise       
  commande pourrait être confirmée. → Stocker l'order_id dans les métadonnées du PaymentIntent Stripe et l'utiliser pour
   la correspondance.

  [BAS] Le prix des items dans CreatePaymentIntentRequest (schémas) est fourni par le frontend et utilisé tel quel pour 
  le calcul du total. Bien que le total soit calculé côté serveur, les prix unitaires proviennent du client sans
  validation contre le menu réel en base de données. Un client malveillant pourrait envoyer price=0.01 pour tous les    
  items. → Valider les prix contre le menu_data en base lors de la création du PaymentIntent.

  d. Évaluation de la Sécurité

  🔴 CRITIQUE

  [CRITIQUE-1] JWT non vérifié sur /api/v1/auth/me                                                                      
   
  auth.py:159 appelle _decode_jwt_payload(token) au lieu de _verify_jwt(token). La fonction _decode_jwt_payload décode  
  le payload sans vérifier la signature RS256. Tout attaquant peut construire un JWT arbitraire avec n'importe quel sub
  (user_id) et accéder aux informations d'un autre utilisateur.                                                         
                                                            
  # Actuel — DANGEREUX
  payload = _decode_jwt_payload(token)  # pas de vérification de signature
                                                                                                                        
  # Corrigé
  payload = _verify_jwt(token)  # utilise JWKS Clerk                                                                    
                                                                                                                        
  [CRITIQUE-2] Modification non authentifiée du profil restaurant
                                                                                                                        
  PATCH /api/v1/restaurants/{slug} (restaurants.py:120) ne requiert aucune authentification. N'importe qui peut :       
  - Modifier le nom, l'adresse, les horaires d'un restaurant
  - Changer le stripe_account_id pour rediriger tous les futurs paiements vers son propre compte Stripe Connect         
                                                                                                               
  Ce vecteur représente un risque financier direct et immédiat. Le même problème affecte POST /{slug}/logo et POST      
  /onboarding/complete.                                                                                                 
                                                                                                                        
  [CRITIQUE-3] Aucun rate limiting sur les endpoints IA                                                                 
                                                            
  POST /api/public/menus/{slug}/chat et /chat/stream n'ont aucune limitation de débit. Un attaquant peut épuiser le     
  quota Gemini API en envoyant des milliers de requêtes, générant des coûts massifs ou rendant le service indisponible
  pour tous les utilisateurs.                                                                                           
                                                            
  🟠 ÉLEVÉ                                                                                                              
   
  [ÉLEVÉ-1] Reçu PDF accessible sans authentification                                                                   
                                                            
  GET /api/v1/payments/{payment_intent_id}/receipt.pdf (payments.py:423) : tout utilisateur connaissant un              
  payment_intent_id (format pi_...) peut télécharger le reçu d'un autre client, exposant son montant, ses articles
  commandés, et l'horodatage. Les payment intent IDs sont visibles dans les URLs de la page ThankYou.                   
                                                            
  [ÉLEVÉ-2] stripe_account_id exposé dans la réponse publique                                                           
   
  GET /api/v1/restaurants/{slug} retourne le stripe_account_id (un identifiant Stripe Connect acct_...) dans la réponse 
  publique. Ce champ ne devrait pas être exposé aux clients.
                                                                                                                        
  [ÉLEVÉ-3] actor_id hardcodé dans les logs d'audit admin                                                               
   
  admin.py:148 logue actor_id="superadmin" (chaîne littérale) au lieu de l'identifiant réel de l'administrateur extrait 
  du JWT. La piste d'audit ne permet pas d'identifier quel admin a effectué l'action.
                                                                                                                        
  🟡 MOYEN                                                  

  [MOYEN-1] En-têtes de sécurité HTTP absents                                                                           
   
  Aucun des middlewares FastAPI ne pose de headers de sécurité : Content-Security-Policy, X-Frame-Options,              
  X-Content-Type-Options, Strict-Transport-Security. → Ajouter un middleware de sécurité (ex: secure library pour
  FastAPI).                                                                                                             
                                                            
  [MOYEN-2] Content-Type header des uploads non validé par magic bytes                                                  
   
  upload_logo (restaurants.py:167) vérifie file.content_type (header HTTP contrôlé par l'attaquant) avant de lire les   
  bytes. Bien que PIL soit robuste, la validation côté client contourne un filtre incorrect. → Toujours utiliser
  detect_mime_type() sur le contenu réel (magic bytes) comme vérification primaire.                                     
                                                            
  [MOYEN-3] allow_credentials=True avec allow_methods=["*"]                                                             
   
  La configuration CORS (main.py:108-114) combine allow_credentials=True et allow_methods=["*"]. Selon la configuration 
  CORS_ORIGINS, cela pourrait permettre des requêtes cross-origin authentifiées avec des méthodes non standard. →
  Restreindre à allow_methods=["GET", "POST", "PATCH", "DELETE"].                                                       
                                                            
  [MOYEN-4] Validation Svix avec secret vide = bypass en dev                                                            
   
  _verify_svix_signature() (auth.py:204) : si secret="", la fonction retourne True sans vérification. En cas de mauvaise
   configuration (variable non définie en production), tous les webhooks Clerk passeraient sans vérification. → Logger
  un warning explicite au démarrage si CLERK_WEBHOOK_SECRET est vide et que IS_PRODUCTION=true.

  e. Performance et Fiabilité

  Points forts :
  - OCR avec retry exponentiel (3 tentatives) et cache Redis 24h par SHA256 du PDF
  - BackgroundTasks FastAPI pour les emails et les notifications KDS (non-bloquant)                                     
  - Menu public mis en cache Redis (5 min TTL) avec invalidation explicite à chaque édition
  - Logging structuré des requêtes lentes (>500ms)                                                                      
  - Workbox runtime caching côté PWA pour les assets statiques et l'API publique                                        
                                                                                                                        
  Problèmes identifiés :                                                                                                
                                                                                                                        
  [ÉLEVÉ] asyncio.run() dans un contexte async (payments.py:294)                                                        
                                                            
  La fonction _broadcast_order_confirmed est appelée via background_tasks.add_task(). FastAPI exécute les background    
  tasks dans un thread pool lorsqu'elles sont synchrones, mais asyncio.run() dans un thread qui partage l'event loop
  peut lever RuntimeError: This event loop is already running selon la configuration d'Uvicorn. → Rendre                
  _broadcast_order_confirmed asynchrone ou utiliser asyncio.run_coroutine_threadsafe().

  [MOYEN] N+1 dans list_restaurants (admin.py:109)                                                                      
   
  Toutes les subscriptions sont chargées en mémoire puis filtrées en Python (sub_map.get(menu.slug)). À l'échelle (des  
  centaines de restaurants), une jointure SQL serait plus efficace.
                                                                                                                        
  [MOYEN] Pas de pagination côté list_restaurants admin                                                                 
   
  L'endpoint /api/v1/admin/restaurants charge tous les menus sans LIMIT/OFFSET. → Ajouter limit/offset comme pour       
  audit-logs.                                               
                                                                                                                        
  [BAS] Streaming chat bloque si Gemini est lent                                                                        
   
  Le streaming SSE dans public.py est synchrone (itération sur generate_content_stream). Avec la configuration par      
  défaut, un timeout de 30s Uvicorn peut expirer avant la fin du stream sur des menus complexes. → Configurer
  --timeout-keep-alive et envisager un timeout explicite par chunk.

  Points forts :
  - AuditLog table dédiée avec actor_type, actor_id, action, resource_type, resource_id, payload, ip_address
  - Webhooks Clerk enregistrent user.created, user.updated, user.deleted — traçabilité RGPD des cycles de vie

  - Service audit_service.py avec helpers typés (log_menu_action, log_payment_action, etc.)
  - .env.example documenté et .gitignore correct (.env exclu)                                                           
                                                                                                                        
  Problèmes identifiés :                                                                                                
                                                                                                                        
  [ÉLEVÉ] AuditLog.actor_id = "superadmin" hardcodé (déjà mentionné)                                                    
                                                            
  [MOYEN] Absence de politique de rétention des données de conversation                                                 
                                                            
  Les conversations chat (Conversation, ChatSession) n'ont pas de TTL ou de politique d'archivage définie. Pour la      
  conformité RGPD (droit à l'oubli), ces données doivent être effaçables sur demande ou après une durée maximale.
  L'endpoint DELETE /conversation existe côté client mais n'est pas déclenché automatiquement.                          
                                                            
  [MOYEN] Feedback NPS stocké sans pseudonymisation                                                                     
   
  AuditLog.payload pour feedback.nps peut contenir des commentaires libres potentiellement identifiants. Bien que le    
  payment_intent_id soit une référence indirecte, le commentaire texte peut contenir des informations personnelles. →
  Documenter la politique de rétention dans la DPA (Data Processing Agreement).                                         
                                                            
  [BAS] Pas de validation de format pour les dates dans audit-logs

  L'endpoint GET /api/v1/admin/audit-logs parse from_date/to_date avec un simple try/except ValueError qui ignore       
  silencieusement les dates invalides plutôt que de retourner un 400.
                                                                                                                        
  [BAS] KDS_SECRET_TOKEN non audité                                                                                     
   
  Les accès au KDS ne génèrent pas d'entrée dans AuditLog. Si le token est compromis, il n'y a aucune traçabilité des   
  consultations de commandes en cuisine.

  EASY.Q présente une architecture backend solide, une couverture de tests sérieuse (416 tests), et une bonne hygiène de
   code. La stratégie de mise en cache Redis, l'audit log RGPD, et l'intégration Stripe sont bien conçus.               
                                                                                                                        
  Toutefois, les 3 vulnérabilités critiques identifiées (modification non authentifiée de profil restaurant avec        
  redirection Stripe possible, JWT non vérifié, absence de rate limiting IA) rendent le déploiement en production
  inacceptable sans correction préalable. Ces problèmes peuvent être résolus en 1 à 2 jours de développement ciblé.     
                                                            
  Après correction des items CRITIQUES et ÉLEVÉS (items 1-7 du tableau), la plateforme atteindrait un niveau de maturité
   suffisant pour un lancement beta contrôlé, avec les items MOYENS à traiter en sprint suivant avant la mise en
  production publique à grande échelle.                                                                                 
                                                            
  Prochaines étapes recommandées :
  1. Corriger les 7 items CRITIQUES/ÉLEVÉS (estimation : 2 jours)
  2. Ajouter un middleware de sécurité HTTP (helmet-equivalent)                                                         
  3. Implémenter le rate limiting via SlowAPI ou Redis         
  4. Faire une revue de la politique de rétention RGPD avec un DPO                                                      
  5. Passer un audit de pénétration externe avant le lancement public
